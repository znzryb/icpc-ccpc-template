# RNG benchmark results

**机器**：Apple M5，macOS 26.2 (build 25C56)
**编译器**：Homebrew GCC 15.2.0（`g++-15`）
**编译 flag**：`-std=c++20 -O2`
**Seed**：12345（三路共享）
**时间**：2026-05-01 18:24 CST

## 测试 1：速度（ns/call）

每轮 N = 100,000,000 次调用，共 5 轮，95% CI 用 Student's t 估（df=4）。

| RNG | ns/call | 95% CI | 相对裸 mt19937_64 |
|-----|--------:|-------:|------------------:|
| SWB + splitmix64 | 3.97 | ±0.12 | 2.886x |
| mt19937_64 + splitmix64 | 1.58 | ±0.06 | 1.146x |
| mt19937_64（裸） | 1.38 | ±0.05 | 1.000x |

## 测试 2：基础统计

每路 N = 10,000,000 个输出。
PASS 阈值：bit 偏差最大值 < 4√N ≈ 12649；byte 卡方最大值 < 330.5（df=255, p<0.001）；|lag-1 z| < 4。

| RNG | bit 最大偏差 | byte 卡方（8 路最大） | lag-1 z | 判定 |
|-----|------------:|--------------------:|--------:|-----|
| SWB + splitmix64 | 3526 | 323.07 | 0.173 | PASS |
| mt19937_64 + splitmix64 | 3680 | 284.91 | -0.054 | PASS |
| mt19937_64（裸） | 4860 | 317.27 | -1.518 | PASS |

### 三个指标具体在测什么

**列 1：bit 最大偏差** — 把 N=10⁷ 个 64-bit 输出的每一个 bit 位独立看：第 0 位上 N 个输出里有几次取 1？理想公平 RNG 应该是 N/2 = 500 万次。**最大偏差** = 64 个 bit 位置里"距离 500 万最远的那个"的距离。理论 σ = √(N/4) ≈ 1581，所以"距离 500 万 1500 上下"是合理范围。三路最大偏差 3526 / 3680 / 4860，全在 3σ 量级，远低于 4√N ≈ 12649 的 PASS 阈。如果某条 bit lane 整个偏死（比如 bit 0 永远是 0），这一列会爆炸。

**列 2：byte 卡方（8 路最大）** — 把 uint64 拆成 8 个字节流（最低字节、次低字节、…… 最高字节），每流里 256 个可能值各出现多少次？理想均匀分布每桶 N/256 = 39062.5 次。对每流跑一次 χ²（自由度 255），取 8 流里**最大的那个 χ²**。阈值 330.5 = χ²₂₅₅ 的 99.9% 上分位数（p < 0.001）。SWB+splitmix64 的 323.07 离 330.5 只差 7、有点贴边但仍是 PASS；其他两路也都过。**这一列查的是某个字节流是否有"某些字节值出现频率系统性偏多/偏少"的偏差**。

**列 3：lag-1 z** — 相邻两次输出 v[i] 和 v[i-1] 异或一下，数它有几个 1。理想公平 RNG 上 popcount(v[i] ⊕ v[i-1]) 的期望是 32，N-1 对加起来期望 32(N-1)。把 (实测 - 期望) 除以标准差 4√(N-1)，得 z-score。|z| < 4 ⟺ 在 4σ 内 ⟺ PASS。三路 0.173 / -0.054 / -1.518 都在 ±2σ 内。**这一列查相邻输出是否过于相似（z 偏正）或过于反相（z 偏负）**——比如线性同余生成器低位常有 lag 相关。

总的来看，三列都是"如果 RNG 实现层面有低级 bug 会被这一列抓住"的兜底测试。三路全 PASS 只能说 splitmix64 包装没把输出写错，**不**能反向证明 RNG 抗得住攻击——所以才要测试 3 的 BM 攻击。

## 测试 3：F₂ 上的 Berlekamp–Massey 攻击

每路 RNG × 每条 bit lane：取 N_total = 100,000 个输出，抽一条 bit lane，前 50,000 bit 喂给 F₂ 上的 BM 训练，后 50,000 bit 验证。

- **复盘度数 d** = BM 找到的最小多项式度数。mt19937_64 在 F₂ 上的线性复杂度恰为 19937。
- **命中率** 在验证半段：100% 表示 LFSR 完美预测（RNG 被破）；~50% 表示 BM 只是在训练段过拟合噪声、对验证段无能为力（RNG 抗住）。
- 安全 RNG 上 d 应饱和到训练长度的一半 = 25,000。

| SWB + splitmix64 | 7 | 25002 | 0.4987 | 0.036 | 抗住 |

## 结论

**1. BM 攻击实证完全符合预期**：裸 `mt19937_64` 在 5 个 bit lane 上一致复盘出 degree = 19937 的最小多项式、验证半段 100% 命中——这就是 Sugar_fan blog 描述的"线性结构 + Berlekamp–Massey 即破"的现场。两个 splitmix64 包装版的 BM 度数都饱和在 25,000（训练长度的一半，BM 在非线性流上的标准失败模式），命中率锁在 50% ± 0.5%。**splitmix64 的非线性混合对挡 F₂ 线性 BM 攻击足够有效**——不仅 SWB 这一路抗住，连原本被破的 mt19937_64 套上 splitmix64 也救回来了。

**2. 统计指标三路全 PASS**：bit balance、byte 卡方、lag-1 自相关都没异常。这与 BM 测试是互补维度——统计随机性和"抗结构性预测"是两回事，前者只能排除最低层的实现 bug。

**3. 速度上一个反预期**：模板文本（引自 CF 测试，Linux 平台）声称裸 SWB ≈ 1.77 ns/call、比裸 mt19937_64（2.74 ns/call）快约 35%。但本机 Apple M5 + Homebrew GCC 15.2 上：

- 裸 mt19937_64 = **1.38** ns/call（比 CF 数字快 2 倍，arm64 + 现代 GCC 帮助大）
- mt19937_64 + splitmix64 = **1.58** ns/call（splitmix64 套壳成本仅 +0.20 ns）
- SWB + splitmix64 = **3.97** ns/call ± 0.12（**反而比 mt19937_64 慢 ~2.5×**）

**怀疑的根因**：Homebrew GCC 15.2 的 libstdc++ 对 `subtract_with_carry_engine<uint64_t, 64, 5, 12>` 没做和 mt19937_64 同等程度的内联/向量化优化（mt19937_64 在 GCC 里是高度调过的特化）；macOS arm64 下还有不同的内存模型/缓存行为也可能有影响。**模板里"SWB 比 mt19937_64 快"的速度论点在本机不成立**——但安全性论点（这才是换 RNG 的真正动机）成立。

**4. 给模板的回写建议**（如有需要）：

- 安全性那段不动，本次 BM 实测正向佐证
